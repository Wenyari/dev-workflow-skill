# 6月23日人机协作开发工作流通用准则

## **1\. 核心理念**



本准则用于指导 Human 与 Agent 协作完成软件开发工作。核心原则是 Human design, Agents code：Human 负责业务判断、目标取舍和最终放行，Agent 负责上下文整理、方案生成、代码实施、检查报告和风险提示。



这套流程不是把所有需求都塞进完整文档链路，而是先判断复杂度，再选择合适的协作强度。简单问题快速处理，复杂问题分阶段确认。



基本原则：

- 从小而明确的任务开始。

- Agent 每次改动前必须复述计划并等待 Human 确认。

- Human 保留业务目标、范围、权限、接口语义、体验验收的最终决策权。

- Agent 必须主动指出需求、设计、接口、代码之间的不一致。

- 代码事实优先于文档描述。

- 文档是可审核产物，不是最终事实。

- 没有必要时不创建流程产物。

## **2\. 决策树**



所有需求先走决策树，再决定采用 L0、L1、L2 或 L3。



```Plain Text
flowchart TD
    A["收到需求"] --> B{"是否仅涉及文案、样式微调、单字段或明显 bug？"}
    B -->|是| L0["L0 快速协作"]
    B -->|否| C{"是否是已有页面的小功能增强或单组件改动？"}
    C -->|是| L1["L1 轻量流程"]
    C -->|否| D{"是否是新页面、Figma 还原、多接口或页面级状态？"}
    D -->|是| L2["L2 标准流程"]
    D -->|否| E{"是否涉及权限、跨页面联动、公共组件、复杂状态或架构调整？"}
    E -->|是| L3["L3 完整流程"]
    E -->|否| L1
```



判定原则：



- 能 L0 解决的问题，不走 L1。

- 能 L1 解决的问题，不走 L2。

- L2 是新页面和标准页面开发的主流程。

- L3 只用于高风险、高影响、跨边界工作。

- Human 可以手动升档或降档，但 Agent 必须说明风险。

## **3\. 复杂度分级**



|档位|适用场景|推荐流程|Human 参与度|Agent 参与度|是否需要落文档|
|---|---|---|---|---|---|
|L0 快速协作|文案、CSS spacing、单字段展示、明显 bug、单文件小修|读代码 \-\> 复述计划 \-\> 修改 \-\> 验证 \-\> 摘要|20%|80%|否|
|L1 轻量流程|单组件、单接口、小交互、已有页面局部增强|Mini Intake \-\> Mini Plan \-\> 修改 \-\> 验证|35%|65%|通常不需要|
|L2 标准流程|新页面、列表表单、Figma 视觉落地、多接口页面|Context Intake \-\> page\-tech \-\> contract\-check \-\> page\-build \-\> figmaSync|45%|55%|需要|
|L3 完整流程|权限复杂、跨页面联动、公共组件改动、复杂状态、架构调整|九阶段完整流程 \+ 回滚规则 \+ review 节点|55%|45%|必须|



升档条件：



- 接口数量达到 5 个独立接口，至少 L2。

- 表单字段达到 15 个，或表格列达到 12 个，至少 L2。

- 涉及按钮级权限、字段级权限，至少 L2。

- 页面状态达到 4 类且存在双向流转，进入 L3。

- 与上下游页面存在数据回写、跳转传参，进入 L3。

- 修改公共组件、公共请求封装、全局状态、路由基础设施，进入 L3。

不建议加入“团队熟悉度”作为自动升降档判据。它是主观因素，可以作为 Human 手动升档理由，但不应写入机械判定规则。



## **4\. 各档流程**



### **4\.1 L0 快速协作**



适用：



- 文案修正。

- 简单样式调整。

- 单字段展示变更。

- 明显 bug 修复。

- 不改变接口、状态、路由、组件边界的小改动。

流程：



```Plain Text
Agent 读取相关文件
Agent 复述最小改动计划
Human 确认
Agent 修改
Agent 运行必要验证
Agent 输出变更摘要
```



禁止：



- 不创建 page\-tech。

- 不创建 contract\-report。

- 不创建 foundation\-summary。

- 不顺手重构。

- 不扩大到相邻模块。

### **4\.2 L1 轻量流程**



适用：



- 已有页面增加一个小交互。

- 已有组件补一个 props。

- 单接口接入。

- 小范围空态、错误态、loading 态补齐。

流程：



```Plain Text
Context Mini Intake
Mini Plan
Human 确认
Code Change
Verify
```



Mini Plan 必须包含：



- 要改什么。

- 涉及哪些文件。

- 不改什么。

- 验证方式。

L1 的计划可以保留在对话中，不强制落 Markdown。



### **4\.3 L2 标准流程**



适用：



- 新页面。

- 标准列表、详情、弹窗表单。

- 有 Figma 视觉落地要求。

- 有多个接口但业务状态不复杂。

流程：



```Plain Text
Context Intake
page-tech
contract-check
page-build
foundation-freeze
figmaSync plan
figmaSync apply
Human 验收
```



L2 是标准页面开发的主流程，但不等于所有需求的默认流程。



### **4\.4 L3 完整流程**



适用：



- 跨页面联动。

- 复杂权限。

- 字段级控制。

- 复杂状态流转。

- 修改公共组件。

- 修改全局状态。

- 修改架构性能力。

九阶段完整流程：



```Plain Text
flowchart LR
    A["业务目标定义"] --> B["Context Intake"]
    B --> C["需求澄清与范围裁剪"]
    C --> D["page-tech"]
    D --> E["contract-check"]
    E --> F["page-build"]
    F --> G["foundation-freeze"]
    G --> H["figmaSync plan"]
    H --> I["figmaSync apply 与验收"]
```



|阶段|核心目标|主导方|Human 参与度|Agent 参与度|Human 确认|
|---|---|---|---|---|---|
|业务目标定义|明确为什么做、给谁用、成功标准|Human|90%|10%|是|
|Context Intake|收集 PRD、接口、设计、代码上下文|Agent|40%|60%|是|
|需求澄清与范围裁剪|决定做什么、不做什么|Human|70%|30%|是|
|page\-tech|生成页面级技术方案|Agent|35%|65%|是|
|contract\-check|检查方案是否可落地|Agent|20%|80%|是|
|page\-build|创建页面基建文件|Agent|25%|75%|是|
|foundation\-freeze|冻结后续可修改范围|Human \+ Agent|50%|50%|是|
|figmaSync plan|生成视觉实施计划|Agent|30%|70%|是|
|figmaSync apply 与验收|实施视觉并验收体验|Agent 执行，Human 验收|45%|55%|是|



## **5\. 阶段产物契约**



### **5\.1 Context Intake**



输出：



- 业务目标。

- 页面或模块范围。

- 用户角色。

- 字段信息。

- 接口信息。

- 交互规则。

- 技术约束。

- 缺失上下文和待确认项。

要求：



- 不补造接口字段。

- 不补造权限规则。

- 不把设计稿视觉层级直接等同于代码组件层级。

### **5\.2 page\-tech**



输出：



- `page-tech.md`。

内容必须包含：



- 页面范围。

- 路由或入口规划。

- 数据流。

- 状态变化。

- 组件拆分。

- 接口映射。

- 错误态、空态、loading 态。

- 待确认项。

边界：



- 只生成技术方案。

- 不直接创建业务代码。

- 可以建议文件路径，但不能把建议当成已存在事实。

### **5\.3 contract\-check**



输出：



- `contract-report.md`。

实现形态：



- v1 采用人工 checklist 模式。

- Agent 按检查项逐条核对，输出 Markdown 报告。

- 每条结论必须引用真实文件路径、代码片段或文档章节作为证据。

- 后续可以把命名、文件存在性、类型导出等机械项替换为脚本检查。

检查项：



- 入口路径是否符合项目规则。

- 目录和文件命名是否符合项目规则。

- 每个接口是否同时具备接口名、入参、响应、调用组件、触发交互。

- 每个页面状态是否具备来源、变化条件、消费组件。

- 每个组件是否能映射到页面功能。

- 待确认项是否阻塞 page\-build。

边界：



- 只检查一致性。

- 不创建文件。

- 检查失败时回到 page\-tech 修正。

### **5\.4 page\-build**



输出：



- 入口文件。

- 局部组件。

- service 占位。

- types。

- constants。

- 必要导出。

要求：



- 只做基建占位。

- 组件必须有明确 props 类型。

- service 必须有明确函数签名。

- 类型定义必须能支撑后续接口联调。

- 可以加入 `TODO(figmaSync)` 标记视觉待落地位置。

- 可以加入 `TODO(api)` 标记接口待联调位置。

禁止：



- 不写复杂业务逻辑。

- 不写最终视觉样式。

- 不擅自创建全局状态。

- 不修改自动生成文件。

### **5\.5 foundation\-freeze**



`foundation-summary.md` 不是事实本身，而是代码事实的机器生成快照。



输出：



- `foundation-summary.md`。

要求：



- 必须由脚本或 Agent 扫描真实目录树、组件 props、service 签名、types 导出后生成。

- 禁止人工手写或手改。

- 每次 figmaSync plan 前必须重新生成或重新扫描。

- 如果快照与上一次不一致，Agent 必须提示 Human 确认影响。

- 文件顶部必须包含：

```HTML
<!-- AUTO-GENERATED. Do not edit. Regenerate from source files. -->
```



内容必须包含：



- 入口文件。

- 组件文件列表。

- service 文件列表。

- types 文件列表。

- constants 文件列表。

- 数据流摘要。

- 状态归属。

- figmaSync 允许修改的文件。

- figmaSync 禁止修改的文件。

### **5\.6 figmaSync plan**



输入：



- Figma URL。

- 真实基建文件。

- 最新 foundation snapshot。

输出：



- `figma-plan.css`。

- 视觉实施计划。

要求：



- 以真实代码文件作为基建事实。

- 组件拆分以已存在文件为准。

- service 签名作为数据流输入。

- types 作为字段映射依据。

- 只补视觉、布局、UI 组件选型和局部展示细节。

禁止：



- 不重新规划路由。

- 不重写 service 契约。

- 不改变核心状态模型。

- 不因为 Figma 图层结构而重拆业务组件。

### **5\.7 figmaSync apply 与验收**



输出：



- CSS、布局和视图层改动。

- 检查结果。

- 验收意见。

- 后续问题清单。

要求：



- Agent 负责实施和检查。

- Human 负责体验验收和最终放行。

- 验收不通过时，回到 figmaSync plan 或更早阶段定位原因。

## **6\. 回滚与增量执行**

### **6\.1 增量执行**

增量执行只追加，不重做已有产物。

适用：

- 新发现的小需求。

- PRD 补充。

- 设计稿局部补充。

- 缺少一个组件或 service 占位。

规则：

- page\-build 只创建缺失文件，不覆盖已存在文件。

- figmaSync plan 只追加新节点计划，不重写已审核章节。

- Agent 必须说明新增内容和不受影响内容。

### **6\.2 回滚执行**



回滚执行会重做指定阶段及其所有下游阶段。



触发场景：



- page\-tech 业务范围变化，回滚到 page\-tech。

- contract\-check 发现路由、接口、类型不一致，回滚到 page\-tech。

- figmaSync apply 发现基建缺失，回滚到 page\-build 增量执行。

- 验收发现体验方向错误，回滚到 figmaSync plan。

回滚记录：



- 起点阶段。

- 回滚原因。

- 影响范围。

- 需要重做的产物。

第一版可以用人工记录，不强制实现 hash。后续可引入文件 mtime、size 或内容 hash 做阶段一致性校验。



## **7\. 最佳实践**



### **7\.1 代码事实优先**



真实代码是事实，文档是意图或快照。page\-tech 是方案，不是事实；foundation\-summary 是机器生成快照，也不是事实本身。



### **7\.2 先判断档位，再执行流程**



不要默认进入完整流程。每次需求先判断 L0、L1、L2、L3，再决定是否需要文档产物。



### **7\.3 一次只做一类变更**



文档整理、契约检查、基建创建、视觉计划、CSS 实施、问题修复不要混在一次改动里。



### **7\.4 Human 保留关键决策权**



以下内容必须由 Human 确认：



- 业务目标。

- 需求范围。

- 权限规则。

- 接口语义。

- 状态机。

- 目录和文件创建清单。

- 是否允许修改公共组件。

- 是否允许引入新的全局状态。

- 最终体验验收。

### **7\.5 Agent 必须主动指出风险**



Agent 发现以下问题时必须明确指出：



- 需求和接口不一致。

- 设计稿无法映射到当前业务结构。

- 状态不应该进入全局状态。

- 组件拆分过细或过粗。

- CSS 方案会污染公共样式。

- 需要修改公共模块但影响范围未确认。

- 用户方案会导致后续维护成本过高。

### **7\.6 Code Review 节点**



L2 和 L3 必须在实现后增加 review 视角：



- 是否越过本阶段边界。

- 是否修改了未授权文件。

- 是否引入全局状态或公共组件影响。

- 是否缺少必要验证。

- 是否存在文档和代码事实不一致。

## **8\. 提效效果**

主要收益不是减少输入字数，而是降低返工成本。

建议跟踪的指标：

- 需求澄清轮次。

- 首次可运行页面耗时。

- 因结构错误导致的返工次数。

- Code Review 问题数量。

- Figma 还原返工次数。

- 接口联调前类型问题数量。

- 上线前缺陷数量。

预期效果：

- 小需求不被流程压垮。

- 新页面有稳定产物链路。

- 视觉还原不重拆业务结构。

- 接口、类型、状态在早期对齐。

- review 更聚焦。

- 复杂需求可以定位到具体阶段回滚。

## **9\. 推荐执行口径**



```Plain Text
先判断需求档位。
L0/L1 走轻量协作，保留改动计划和验证结果即可。
L2 走标准页面流程，产出 page-tech、contract-report、基建和视觉计划。
L3 走完整九阶段，并启用回滚与 review 节点。
Human 决策业务和验收，Agent 生成、实施、检查和提示风险。
```

最终目标

建立一套**可控、可审查、可提效的人机协作开发系统**。它要解决的核心问题是：不同经验的人面对同一份需求、设计稿和代码仓库时，产出质量差异太大；AI 如果没有边界，也会把需求、方案、视觉、代码混在一起做，短期快，长期返工多。

几个典型场景：

最终目标可以这样定义：

> 让 Human 专注业务判断、体验验收和关键取舍，让 Agent 承担上下文整理、方案生成、基建创建、视觉实施和质量检查；通过复杂度分级和阶段产物，把 AI 开发从“不可控的一次性生成”变成“可审查、可回滚、可复用的协作流程”。
> 
> 

最终效果：

- 新手也能按稳定路径完成页面开发。

- 老手不会被过度流程拖慢。

- 小修复走轻量流程，新页面走标准流程，复杂需求走完整流程。

- AI 不再直接越权决定业务结构。

- Figma 视觉不会反向污染代码骨架。

- 文档、代码、设计之间有明确契约。

- review 从“看一大坨改动”变成“按阶段验收产物”。

- 返工减少，页面交付更稳定，团队产出质量更一致。
