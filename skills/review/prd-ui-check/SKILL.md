# PRD 与 UI 检查

## 对齐边诊断

| 项 | 值 |
|---|---|
| 服务对齐边 | PM↔UI |
| 现状分级 | 主痛 |
| AI 形态 | 补契约工具 |
| 主战场 | UI 评审前 12-24 小时 |

## 这个 skill 解决什么问题

在 UI 评审前，由人确定页面与 Figma 节点范围，批量检查 PRD 与 UI 的业务语义是否一致，提前发现 UI 未表达、UI 越界、双方冲突和证据缺口。

## 什么时候用

- 用户显式输入 `$prd-ui-check <prd>`。
- 用户要求“检查 PRD 和 UI 是否一致”“看看 UI 有没有漏需求”。
- 用户准备 UI 评审，并会提供页面、弹窗或交互态的 Figma 节点链接。

## 前置产物

| 产物 | 来源 | 是否必需 |
|---|---|---|
| PRD 全文 | 本地 Markdown / 文本 / `lark-read` 结果 | 必需 |
| 当前页面或状态的 Figma 节点链接 | 用户按页面提供 | 必需 |
| Figma MCP 访问能力 | Figma MCP | 必需，不使用截图 OCR 替代 |

## 输出产物

| 产物 | 位置 | 下游消费者 |
|---|---|---|
| `prd-ui-check-report.md` | PRD 所在目录或用户指定目录 | UI 评审参与者；后续未闭环清单 |

## 下一步

- 报告中的差异由 PM 与 UI 负责人确认，不由 Agent 自行取舍。
- 评审后仍未解决的问题可进入 `artifact/open-issues`。
- 误报、漏报及人工修正结果可进入 `artifact/calibration`。

## 明确不做

- 不审查 PRD 自身是否完整、合理或具备技术可行性；这属于 `devFlow prd-review`。
- 不做像素、颜色、字号、间距、圆角和视觉还原检查；这属于 `figmaSync`。
- 不读取前端仓库，不判断 UI 应如何编码，也不生成 CSS。
- 不主动扫描整个 Figma 文件；检查范围由用户按页面、状态指定。
- 不把尚未提供的节点判定为 UI 遗漏；用户确认当前页面节点已提供完成后，才允许形成“UI 未表达”结论。

## 命令路由

### `$prd-ui-check <prd>`

用途：启动一次 PRD 与 UI 的批量语义对齐检查。

进入命令后立即读取 [references/check.md](./references/check.md)，严格按其中的阶段推进。不要要求用户一次性提供整份 Figma，也不要把检查过程拆成 `start`、`add-node`、`finalize` 等额外子命令。

报告模板：[templates/prd-ui-check-report.md](./templates/prd-ui-check-report.md)。

## 通用原则

- 使用中文输出。
- PRD 是需求事实来源，Figma 节点是 UI 表达事实来源；双方冲突时只列证据，不代替 PM 或 UI 负责人决策。
- 节点语义必须区分“可直接观察的事实”和“根据名称、层级、位置、变量推测的语义”。
- 默认批量记录节点事实和差异，不要求用户逐节点确认；仅当歧义会阻断后续判断时提问。
- 用户主动提供的人工检查结果优先于节点名称推测，并在最终报告中标记为“人工确认”。
- 全部指定范围检查完成后，一次性输出差异、证据不足和已确认一致项清单。
- 每条差异必须引用 PRD 章节或行号，并引用一个或多个 Figma 节点；UI 未表达项必须额外记录用户确认完成的页面范围。
- 不建立正则驱动的通用规则库。语义判断由 Agent 完成，人工确认负责校正。
- 不自动生成 P0 / P1 / P2。报告描述影响，由评审参与者确定优先级。

## 输入约束

- 本地 PRD：直接读取，引用使用相对路径与行号或章节标题。
- 飞书 PRD：先使用 `lark-read` 获取全文，再开始检查；读取失败时停止。
- PDF、图片或聊天记录：先转换为结构稳定的 Markdown 或文本，不基于 OCR 结果直接出最终结论。
- Figma 链接必须包含用户指定的节点范围；只有文件级链接时，请用户补充当前页面节点链接。

## 交付约束

- 默认输出 `prd-ui-check-report.md`，落盘前确认存放目录。
- 已有同名报告时不覆盖，先询问用户改名或明确覆盖。
- 用户尚未声明全部页面检查完成时，仅在用户主动询问时输出简短进度，不提前输出完整问题清单，也不生成最终报告。
- 最终报告必须保留检查范围、已确认一致项和证据不足项，不能只列问题。
