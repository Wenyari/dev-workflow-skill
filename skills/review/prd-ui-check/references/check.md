# PRD 与 UI 多轮检查规则

## 目标

以 PRD 为需求事实来源，以用户逐轮提供的 Figma 节点为 UI 表达事实来源，建立可追溯的语义映射。只检查双方表达是否一致，不检查视觉还原和技术实现。

## 阶段一：读取 PRD

读取 PRD 全文后，按页面或功能模块整理“待对照需求点”。每个需求点只提取可由 UI 表达或影响 UI 行为的信息：

- 页面、模块、入口和跳转关系。
- 用户角色、权限和可见性。
- 业务对象、字段、字段含义和必填规则。
- 操作、触发条件、确认步骤和操作结果。
- 枚举值、状态和状态流转。
- 弹窗、抽屉、提示、成功、失败、空态和禁用态。
- PRD 明确要求的交互规则和业务文案。

不要提取 API、数据库、代码结构、性能、监控、灰度和数据迁移要求。

需求点使用以下内部结构组织，不要求单独落盘：

```text
需求点编号
页面或模块
场景或状态
角色
对象
操作或字段
条件
预期 UI 表达
PRD 引用
```

完成后只向用户展示页面或模块目录和每页需求点数量，不一次性输出所有细节。请用户选择当前页面并提供对应 Figma 节点链接。

## 阶段二：建立页面覆盖范围

每个页面维护以下状态：

- `待提供`：尚未收到对应 Figma 节点。
- `检查中`：已收到部分页面、弹窗或交互态节点。
- `已完成`：用户明确确认该页面相关节点已提供完成。

只有用户可以把页面从“检查中”确认成“已完成”。Agent 可以根据 PRD 提醒可能还需要哪些节点，但不能自行宣布覆盖完成。

## 阶段三：逐个读取 Figma 节点

每次只分析用户当前提供的节点，不主动扫描整个文件。

按以下顺序按需读取：

1. 使用 `get_metadata` 读取节点名称、类型、层级、同级关系、布局方向和位置。
2. 需要补充 Component、Instance、Variant、文本或绑定变量语义时，使用 `get_design_context`。
3. 仅当层级和位置不足以判断区域关系时，使用 `get_screenshot` 辅助确认。
4. 仅当节点绑定的变量名称可能表达业务状态时，使用 `get_variable_defs`；忽略纯颜色、间距、字号和圆角变量。

节点过大时，根据 metadata 选择与当前页面需求相关的子节点继续读取。递归范围由当前需求点限定，不读取无关页面。

### 需要提取的 UI 语义

- 页面、区域、弹窗、抽屉和 Tab 的用途。
- 可见文案、字段、表格列、按钮和入口。
- 父子节点、同级节点和重复结构表达的业务关系。
- Auto Layout、分组和相对位置表达的信息层级。
- Component、Instance、Variant 名称与属性表达的状态。
- 节点显隐、启用、选中和禁用状态。
- 语义化变量名称表达的状态或用途。
- 能够直接读取到的原型跳转或交互关系。

### 必须忽略的信息

- 精确颜色、字号、间距、尺寸、边框、阴影和圆角。
- CSS variable 与项目 Theme 的匹配关系。
- Apex UI、前端组件、Icon 和 assets 的复用方式。
- 响应式实现、代码结构和 CSS 写法。
- 单纯根据视觉风格推测出的业务规则。

### 语义证据优先级

1. 可见文案、Variant 属性和明确交互关系。
2. 节点名称、组件名称和实例名称。
3. 节点层级、同级关系和重复结构。
4. 布局位置、分组和 Auto Layout。
5. 具有业务含义的变量名称。
6. 仅靠视觉外观得到的推测，不进入正式事实。

### 每轮确认格式

读取节点后，先向用户输出：

```text
当前节点
所属页面与场景
直接观察到的 UI 事实
推测的 UI 语义及推测依据
可能对应的 PRD 需求点
当前无法确认的内容
```

请用户确认、修正或继续提供相关节点。用户修正后的内容优先于节点名称推测，并在最终报告中标记为“人工确认”。

## 阶段四：页面级比对

用户确认页面节点已提供完成后，对该页面的需求点逐条比对。

### PRD 已定义但 UI 未表达

满足全部条件才能成立：

- PRD 有明确、可引用的要求。
- 当前页面已由用户确认节点提供完成。
- 已提供节点中没有对应字段、操作、状态或交互表达。

结论使用“UI 未表达”，不要直接写“设计师遗漏”。

### UI 已表达但 PRD 未定义

UI 节点明确出现字段、操作、状态或流程，但 PRD 全文找不到依据。不要判断这是合理扩展还是越界，只提出待 PM 确认的问题。

### PRD 与 UI 冲突

双方都有明确证据，但语义不能同时成立，例如：

- 单选与多选不一致。
- 可见角色或按钮出现条件不一致。
- 枚举值、状态数量或状态名称不一致。
- PRD 要求确认后执行，UI 表达为直接执行。
- 页面跳转目标或操作结果不一致。

必须同时列出双方原始表达，不自行选择采用哪一方。

### 证据不足

以下情况归入证据不足：

- 页面仍处于“待提供”或“检查中”。
- 节点名称、层级或变量只支持推测，用户尚未确认。
- PRD 提到交互态，但用户尚未提供相应节点。
- Figma 静态节点无法表达动态规则或权限条件。

### 已确认一致

双方有明确证据且语义一致时，记录需求点、PRD 引用和 Figma 节点。已确认一致项进入覆盖记录，不进入差异问题列表。

## 阶段五：继续或结束

完成一个页面后，向用户展示该页面的阶段性结论，并请用户选择下一个页面或补充节点。

只有用户明确表示全部页面检查完成后，才进入最终报告。否则保持多轮检查状态，不生成最终文件。

## 最终报告

按 [../templates/prd-ui-check-report.md](../templates/prd-ui-check-report.md) 生成报告。

每条差异必须包含：

- 页面与场景。
- PRD 表达和引用。
- UI 表达和 Figma 节点引用。
- 差异结论与判断依据。
- 置信度：高、中、低。
- 需要谁回答的具体问题。

“UI 未表达”项没有对应 UI 节点时，必须列出用户确认已完成的页面范围和已检查节点，不能伪造 Figma 引用。

## 交付前自检

- 每个“UI 未表达”项所在页面均由用户确认节点提供完成。
- 每个“UI 已表达但 PRD 未定义”项均有 Figma 节点引用。
- 每个冲突项均同时列出 PRD 与 Figma 证据。
- 直接事实与语义推测已经分开，未把低可信推测写成事实。
- 未检查具体样式、前端实现和 PRD 技术可行性。
- 报告包含页面覆盖情况、证据不足项和已确认一致项。
