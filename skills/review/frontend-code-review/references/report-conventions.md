# 报告约定

`code-review-report.md` 的字段格式、证据格式、严重度分档硬约束。

---

## 严重度三档

**只用三档，不用 P0 / P1 / P2**。P0/P1/P2 属于排期决策，由开发和评审者定，Agent 只描述影响。

| 档 | 判定口径 | 示例 |
|---|---|---|
| 🔴 阻塞 | 涉及数据安全、用户无法完成核心操作、必然导致生产事故 | 富文本未做 XSS 过滤；金额用 `parseFloat` 计算；必填字段缺失校验导致空数据入库 |
| 🟡 建议 | 违反规范但不阻塞使用，影响体验或一致性 | 错误文案不符「字段名 + 格式不正确」模板；空状态无操作引导；必填标识 `*` 缺失；只读态样式与规范不符 |
| 🟢 提示 | 规范建议但非强制；可优化项 | 邮箱可支持域名自动补全；手机号可支持国际区号；工具栏可收起高级功能 |

判档参考：

- 涉及安全（XSS、注入、金额精度、权限）→ 🔴
- 违反 `field-common.md` 中「必填标识」「错误提示」「校验触发」「只读状态」明确规定 → 🟡
- 违反 `fields-*` 分册中「UI 与体验要点」「产品要点」提到的强约束 → 🟡
- 违反上述分册中「可支持」「建议」类描述 → 🟢

不确定时**往低档走**，避免虚报阻塞项造成噪音。

---

## 单条问题的字段格式

每条问题必须包含以下字段，缺一不可：

```markdown
#### 问题 <编号>：<一句话描述>

- **严重度**：🔴 / 🟡 / 🟢
- **位置**：`src/pages/foo/Bar.tsx:42-48`
- **违反规范**：`tools/product-design-specs/field-common.md` §错误提示
- **规范原文**：> 必填校验：提示文本 + 字段名，例如"请输入客户名称""请选择客户等级"。
- **证据**：
  ```tsx
  { required: true, message: '不能为空' }
  ```
- **建议**：改为 `message: '请输入客户名称'`，遵循「提示文本 + 字段名」模板。
```

### 字段规则

- **编号**：全报告顺序编号（不按严重度分开编号），便于追溯
- **位置**：`文件:行号` 或 `文件:起始行-结束行`；必须是相对仓库根的路径
- **违反规范**：必须指到具体分册 + 章节（`§章节名`）
- **规范原文**：从分册复制原文，用 `> ` 引用格式；便于开发者不翻分册就能核对
- **证据**：真实代码片段，用带语言标记的代码块；不做修饰、不删注释
- **建议**：改动方向，一句话；**不给完整改写代码**（避免落入自动修复陷阱）

## 合规项记录

**必须记录已确认合规的字段/交互**，不能只列问题——只列问题会让用户误以为「凡未提到的都有问题」。

格式：

```markdown
- ✅ `src/pages/foo/Bar.tsx:80` 手机号字段：正则、位数、错误文案均符合 `fields-text.md` §短文本 手机号规范
- ✅ `src/pages/foo/Bar.tsx:120` 表单错误提示：字段名 + 具体原因，符合 `field-common.md` §错误提示
```

一行一条，简短，不需要证据代码块。

---

## 待确认字段

`field-mapping.md` 中标为「待确认」的字段全部汇总到本板块。格式：

```markdown
- 位置：`src/pages/foo/Bar.tsx:42`
- 代码片段：`<Input name="mystery" />`
- 已知信号：通用 Input，label「附加信息」
- 推测候选：短文本 / 编码 / 备注
- 缺失信息：需要产品/开发说明业务含义与录入方式
```

**待确认字段不生成任何问题条目**——直到人工补充信息后才能重新判断。

## 去重规则

同一规范条目在多个位置违反时，聚合成**一条**问题，位置列表列出全部出现处：

```markdown
#### 问题 12：错误提示未使用「提示文本 + 字段名」模板

- **严重度**：🟡
- **位置**：
  - `src/pages/contract/Create.tsx:42`
  - `src/pages/contract/Edit.tsx:88`
  - `src/pages/supplier/Form.tsx:120`
- **违反规范**：`tools/product-design-specs/field-common.md` §错误提示
- **规范原文**：> 必填校验：提示文本 + 字段名...
- **证据**（取第一处）：
  ```tsx
  { required: true, message: '不能为空' }
  ```
- **建议**：全部改为 `请输入 <字段名>` / `请选择 <字段名>` 模板。
```

聚合门槛：**同一规范条目**且**违反方式相同**才聚合；同条规范不同违反方式仍分开列。

---

## 不做什么

- 不写「总结」「一句话总评」这类主观段落
- 不写「建议下次注意」这类劝告文案
- 不为每条问题打 P0 / P1 / P2 / 优先级
- 不建议 refactor 方向（如「建议抽公共组件」），只查规范违反
- 不评论代码风格、命名、注释——那是 ESLint / Prettier 和 code review 姿态问题
