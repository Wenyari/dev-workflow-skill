# Review 流程

三阶段：`start` → `batch n` → `finalize`。允许分次执行，中间态落盘。

## start 阶段

### 目标

- 明确 review 范围
- 扫描代码识别涉及的字段类型和交互能力
- 生成批次计划

### 强制步骤

1. **确认 scope**：用户必须提供以下之一
   - 目录路径（例如 `src/pages/contract/`）
   - 文件清单（多行文件路径）
   - PR diff（`git diff main...HEAD` 的输出或 diff 文件）

   scope 不明确时**必须停下问用户**，不要自作主张扩大或缩小范围。

2. **确认工作目录**：批次记录与最终报告的落盘位置。
   - 默认建议：项目根目录下 `.review/frontend/<yyyy-mm-dd>/`
   - 用户可指定其他位置

3. **扫描代码**（只读，不改）：
   - 识别表单、列表、详情、上传、选择器、弹窗、反馈组件
   - 按 [field-mapping.md](./field-mapping.md) 规则把每个字段映射到 `tools/product-design-specs/` 的对应分册
   - 建立「字段清单」：字段名 / 文件:行号 / 推断类型 / 引用规范分册 / 置信度

4. **生成批次计划**：
   - 按文件粒度分批，单批约 500-1000 行代码
   - 相关字段（例如同一表单内的字段）尽量分在同一批
   - 每批标注：批次编号 / 覆盖文件 / 涉及规范分册

5. **落盘**：把「字段清单」和「批次计划」写入工作目录 `plan.md`。

### start 阶段输出

工作目录下必须有：
- `plan.md`：字段清单 + 批次计划

### 明确不做

- 不主动读用户 scope 之外的文件
- 不在 start 阶段做规范比对（那是 batch 的事）
- 不询问用户「这个字段是什么类型」——推断不出来的字段直接标为「待确认」，在 finalize 阶段一次性汇总给用户

---

## batch n 阶段

### 目标

对第 n 批文件按对应规范分册做逐条比对，产出批次中间记录。

### 强制步骤

1. **读取 plan.md** 确认第 n 批的覆盖文件和涉及规范分册。

2. **Lazy load 规范分册**：
   - 只读第 n 批用到的分册
   - 任何字段场景默认先读 `field-common.md`
   - 涉及默认值、空状态、操作反馈、弹窗、文案或中断恢复时读 `interaction.md`
   - **不要一次性加载全部规范**（对齐 `tools/product-design-specs/index.md` 硬约束）

3. **逐条比对**：
   - 对每个字段/交互按分册条目逐项检查
   - 命中问题时记录：`文件:行号` + 违反的规范条目原文 + 代码片段证据 + 建议改动方向
   - 对合规项也要记录（用于最终报告的「已确认合规项」板块）
   - 无法判断的字段标为「待确认」，说明推测依据和缺失信息

4. **严重度分档**（对齐 `report-conventions.md`）：
   - 🔴 阻塞：数据安全（XSS、金额浮点、SQL/HTML 注入面）、用户无法完成核心操作
   - 🟡 建议:违反规范但不阻塞使用（如错误文案模板不符、空状态无引导、必填标识缺失）
   - 🟢 提示:规范建议但非强制（如「可支持自动补全」、「悬浮提示可优化」）

5. **落盘**：把批次结果写入工作目录 `batch-<n>.md`。

### batch 阶段输出

工作目录下：
- `batch-<n>.md`：本批次问题清单 + 合规项 + 待确认字段

### 明确不做

- 不查 ESLint / Prettier / TypeScript 已覆盖的规则（命名、格式、类型）
- 不改代码，不生成 PR
- 不跨批次引用（每批独立可读，避免上下文依赖）
- 不主动扩散到 plan.md 未列出的文件

---

## finalize 阶段

### 目标

合并所有批次记录，输出最终 `code-review-report.md`。

### 强制步骤

1. **确认所有批次已完成**：工作目录下 `batch-1.md` 到 `batch-N.md` 齐全（N 来自 `plan.md`）。
   - 缺批次时**必须停下问用户**：是补跑还是明确跳过。

2. **合并**：
   - 按维度归类问题：字段布局 / 必填校验 / 错误提示 / 只读态 / 列表 / 交互反馈 / 数据安全
   - 按严重度分档（🔴 / 🟡 / 🟢）
   - 去重：同类问题多次出现时聚合成一条，列出全部出现位置

3. **套用模板**：读取 [../templates/code-review-report.md](../templates/code-review-report.md)，按模板填充。

4. **报告结构（五个板块必须齐全）**：
   - 检查范围（scope + 涉及字段类型 + 引用规范分册 + 批次总数）
   - 字段清单
   - 问题清单（按严重度分档）
   - 已确认合规项
   - 待确认字段类型

5. **落盘**：写入用户在 `start` 阶段指定的工作目录，文件名 `code-review-report.md`。
   - 已有同名报告时不覆盖，先询问用户改名或明确覆盖。

### finalize 阶段输出

- `code-review-report.md`

### 明确不做

- 不擅自压缩「已确认合规项」和「待确认字段类型」板块，即使为空也保留标题
- 不自动分配 P0 / P1 / P2
- 不主动创建修复 PR
- 不清理批次中间记录（用户可能需要回顾原始判断依据）
